# Data status — criteria, listings, pins

A single-page tracker for the data backing every priority in the app.
For each criterion this is the source of truth for **what's real, what's
synthetic, and what needs to happen to make it real**. Tick the boxes
as Phase 2 ingestion lands.

This is a follow-on to [`docs/data-sources-uk.md`](./data-sources-uk.md)
which has the per-source ingestion plan. Read that first for citation
detail; come here for the live state.

---

## Status legend

| | Meaning |
|---|---|
| 🔴 | Not started — no pipeline, only synthetic values |
| 🟡 | **WIP / synthetic representation** — the criterion renders, the methodology is authored, but cell values are generated by the seed script not ingested from the real source |
| 🟠 | Real data ingested but not validated — pipeline runs, values land in cells, no spot-check against the publisher |
| 🟢 | Real data ingested and validated against the publisher's reference |

**Today, every criterion is 🟡.** The catalog, transforms, methodology
prose, and WLC engine are all real and correct — but the per-cell
values come from `scripts/generate-london-seed.ts` and
`scripts/generate-west-midlands-seed.ts`, not from EA / Ofsted / NHS /
DEFRA / etc.

Listings, pins, geocoding, and the basemap are a separate matter — see
the listings, pinned-properties, and infrastructure sections below.

### Safety & Hazard

#### flood_risk · Flood risk · 🟡

**Real sources** (from `methodology.ts`):
- [Environment Agency — Flood Map for Planning (Rivers and Sea)](https://environment.data.gov.uk/dataset/8d57464f-d465-11e4-8790-f0def148f590) · OGL v3 · Quarterly · latest Q1 2026
- [EA — Risk of Flooding from Surface Water](https://environment.data.gov.uk/dataset/8a09d10a-2a07-46a8-8a83-7be3a17e29be) · OGL v3 · Annual
- [NRW Flood Map (Wales)](https://datamap.gov.wales/maps/?lang=en&theme=flooding) · OGL v3 · Annual
- [SEPA Flood Maps (Scotland)](https://www.sepa.org.uk/environment/water/flooding/) · OGL v3 · Biennial

**Phase 1.x today**: distance-to-Thames polyline (or per-region rivers)
maps to band; ~4% of cells get a random surface-water `MEDIUM`.

**To go from 🟡 → 🟢:**
- [ ] Pipeline: fetch EA NFHL Flood Zones 2/3 (Wales / Scotland equivalents) as shapefile / WMS
- [ ] Pipeline: clip to active region bboxes; fix invalid geometries via `ST_MakeValid`
- [ ] Pipeline: rasterize Flood Zone 3 → HIGH, Zone 2 → MEDIUM, outside → VERY_LOW
- [ ] Pipeline: overlay Surface Water risk for the `LOW` cluster fill
- [ ] Aggregate: assign band per H3 cell (modal class within cell area)
- [ ] Validate: spot-check 20 random postcodes against EA's online flood checker
- [ ] Replace `rawFloodRisk()` in both region seed scripts with an `external-source-fed` mode

**Caveats**: defended floodplains require manual overlay; climate projection
(UKCP18 sea-level rise) is a separate future criterion.

## Cross-cutting Phase 2 infrastructure

Tasks that are *not* per-criterion but are prerequisites for moving anything
from 🟡 to 🟢:

### Pipeline orchestration
- [ ] Decide: Prefect 2 vs scheduled GitHub Actions (cheap-and-cheerful)
- [ ] Per-source job pattern: fetch → normalise → standardise → write COG + H3 rows → validate → publish
- [ ] Version every dataset (`<criterion>/<yyyy-mm-dd>`); retain last 12 months in S3 / R2

### Storage
- [ ] Postgres + PostGIS + h3-pg for the cell-score table (`suitability.h3_cell_scores`)
- [ ] Object storage (R2 / B2) for COG rasters + PMTiles basemaps
- [ ] Redis for isochrone cache + tile cache

### API surface
- [ ] FastAPI: `/api/v1/suitability/cells` (viewport-bounded), `/api/v1/catalog`, `/api/v1/regions`
- [ ] Tile servers: Martin (vector MVT) + TiTiler (COG → XYZ tiles)
- [ ] Bust caches on dataset version change (hashed URLs)

### Hashed seed URLs (today's hack until full API)
- [ ] Rename `public/data/regions/<id>.cells.json` → `<id>.cells.<hash>.json`
- [ ] Compute hash from `generatedAt` or content; emit alongside the JSON
- [ ] Read hash from a small `index.json` at runtime so the loader can build the right URL

## How to keep this doc accurate

Whenever a criterion crosses a status boundary (🟡 → 🟠, etc.):

1. Update the **status icon** in the per-criterion section + the summary table.
2. Tick the relevant subtask boxes.
3. Update the matching `phase1Estimation` paragraph in
   `src/lib/methodology.ts` to describe the real source rather than the
   synthetic generator.
4. Add a `dataAsOf` field to the seed file metadata so the UI can show
   freshness.
5. Add a one-line note to the relevant phase entry in
   `docs/ARCHITECTURE.md` if the change is meaningful.

A test asserting that *every* criterion has a methodology entry with at
least one non-empty `realSources.url` is a quick win once Vitest lands.
